## 第11章 索引与散列
## 11.1 基本概念
两种基本索引类型
* 顺序索引
* 散列索引

选用索引 考虑的因素：
* 访问类型
* 访问时间
* 插入时间
* 删除时间
* 空间开销

用于在文件中查找记录的属性或属性集 称为 搜索码 search key 

## 11.2 顺序索引
如果包含记录的文件 按照某个搜索码指定的顺序排序，
那么该搜索码对应的索引称为聚集索引 clustering index ， 主索引 primary index

搜索码顺序 和 文件中记录的物理顺序不同的索引 称为 非聚集索引 nonclustering index ，辅助索引 secondary index 

在搜索码上有聚集索引的文件称为 索引顺序文件 index sequential file 


#### 11.2.1 稠密索引和稀疏索引
索引项 index entry  索引记录 index record  由 一个搜索码值 和 指向具有该搜索码值的一条或多条记录的指针组成。

* 稠密索引 dense index ： 每个搜索码都有一个索引项
* 稀疏索引 sparse index ： 只为搜索码的某些值建立索引项

前者查询性能高， 后者维护代价开销小。


#### 11.2.2 多级索引
在原始的内层索引上构造一个稀疏的外层索引。
因为索引项是有序的，所以外层索引可以是稀疏的

多级索引 比 二分搜索 所需的io操作少很多

#### 11.2.3 索引的更新
文件中有记录要插入过删除时，索引都需要更新。



#### 11.2.4 辅助索引
辅助索引必须是稠密索引，每个搜索码都要有一个索引项，对文件中的每条记录都要有指针。

一般来说，辅助索引的结构和聚集索引不同，
聚集索引只指向一条记录即可，其他记录可以通过顺序扫描获得
辅助索引必须包含指向每一条记录的指针

大多数数据库实现在主码上自动创建一个索引，该索引可以用来判断是否重复。
如果没有该索引，每加一条记录就要全量扫描，判断是否重复。

#### 11.2.5 多码上的索引
搜索码可以有多个属性
包含多个属性的搜索码 称为 复合搜索码 composite search key



## 11.3 B+树索引文件
索引顺序文件 最大的缺点，随着文件的增大，索引查找性能和数据顺序扫描性能都会下降。
B+树 在数据 插入和删除的情况下 仍然能保持执行效率。
B+树 采用平衡树， 树根到树叶的每条路径的长度相同。
树中每个非叶节点有 [n/2]~n 个子节点。 n对特定的树是固定的

B+树会增加文件插入和删除处理的性能开销，同时会增加空间开销，但是开销是可以接受的，因为减少文件重组的代价
由于节点可能是半空的，将造成空间浪费，但是考虑到带来的性能的提高，空间开销是可以接受的。

#### 11.3.1 B+树的结构
B+树 是一种多级索引，但是结构不同于多级索引顺序文件
一个节点包含n-1搜索码值 ， n个指针，搜索码值顺序存放
对于 叶子节点 ，指针p指向 搜索码值对应的记录， 指针Pn指向下一个叶子节点

B+树的非叶子节点 形成 叶子节点 上的一个多级稀疏索引
叶子节点和非叶子节点 结构相同，
非叶子节点中的指针 都是指向树中节点的指针。

#### 11.3.4 不唯一的搜索码
如果一个关系 可以拥有 包含同一搜索码值的记录， 那么该搜索码称为 不唯一搜索码
不唯一搜索码的一个问题在于删除效率

解决办法：
唯一化 uniquifier ： 将不唯一的搜索码和另外一个属性结合，形成唯一的搜索码

不唯一搜索码在B+树中存储：
* 每个搜索码存一条记录
* 多个搜索码共用一个桶

## 11.4 B+树扩展

#### 11.4.1 B+树文件组织
B+树文件组织 树的叶子节点存储的是记录而不是指向记录的指针

当使用B+树作为文件组织方式时，空间的利用尤为重要，
通过在分裂和合并时在重新分布中设计更多的兄弟节点，可以改善B+树的空间利用率

B+树索引或 B+树文件组织 ，树中相邻的叶子节点可能分布在磁盘中不同位置。
当一个文件组织最初建立在一组记录上时，可以实现把磁盘中基本连续的块分配给树中连续的叶节点
这样，对叶子节点的顺序扫描就相当于磁盘上几乎是顺序扫描
但是插入删除增多之后， 顺序性逐渐丢失，为了恢复顺序性，需要对索引进行重建。


#### 11.4.2 辅助索引和记录重定位
当B+树的节点分裂后，所有存储了重定位后的记录的指针都要更新
解决方案：辅助索引中，不存储指向被索引的记录的指针，而是存储主索引搜索码属性的值。

大大降低了由文件重组导致的索引更新的代价。
增加了使用辅助索引访问数据的代价。
#### 11.4.3 字符串上的索引
在字符串上创建B+树索引 有两个问题， 字符串变长，字符串可能很长
使用前缀压缩 prefix compression 可以增加节点的扇出。
不存整个搜索码，只存搜索码的前缀

#### 11.4.4 B+树索引的批量加载
将大量数据一次插入到索引中 称为索引的 批量加载 bulk loading 

#### 11.4.5 B树索引文件
B树索引和 B+树索引类似

B树去除了 搜索码值中的冗余

B+树中 搜索码 可能同时存在于 叶子节点 和 非叶子节点。

B树中 搜索码 只会出现在一个节点

在B树中 查询某些搜索码 比B+树快， 有些比B+树慢

B树删除更加复杂，插入略微复杂

B树在空间上的优势不大，通常不能抵消上述不足，大多数数据库采用B+树


#### 11.4.6 闪存
磁盘寻道需要10ms ， 闪存随机块读取在微秒级完成。

闪存唯一缺点，不允许物理层的数据就地更新，
更新操作是 整个闪存块的拷贝加上写操作，块的旧拷贝随后清除，一个块的清除需要1ms
对于闪存， 需要降低块清除次数的索引结构。

B+树索引可以继续在闪存存储上使用，提供可以接受的更新性能，已经比磁盘存储更有效的查询性能。

## 11.6 静态散列
顺序文件组织的缺点是 必须防伪索引结构来定位数据，或者使用二分法搜索，这将导致更多的io操作

散列的文件组织 可以避免访问索引结构

桶：能存储一条或多条记录的存储单位。 通常一个桶就是一个磁盘

散列可以用于两个目的：
* 散列文件组织 hash file organization 
* 散列索引组织 hash index organization 

#### 11.6.1 散列函数
最坏情况是将所有搜索码映射到同一个桶

期望：分布是均匀的 随机的

糟糕的散列函数导致查找所花费的时间 与文件中 搜索码的 数目成正比

#### 11.6.2 桶溢出处理
溢出链：overflow chaining 

* 闭地址 close addressing ， 闭散列 closed hashing ， 上述方式就属于这种
* 开地址 open addressing ： 线性探测法。删除操作麻烦

数据库一般使用 闭地址方案， 因为删除和插入一样重要

#### 11.6.3 散列索引
散列索引将搜索码 和 相应的指针 组织成 散列文件结构

#### 11.7.1 数据结构

#### 11.7.2 查询和更新
#### 11.7.3 静态散列与动态散列比较
