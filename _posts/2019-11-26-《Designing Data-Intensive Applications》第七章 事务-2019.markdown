## 7. Transactions

数据系统很多地方可能出错：
* 硬件故障，软件故障
* 软件崩溃
* 网络中断，导致应用断开数据库
* 多个客户端同时写，覆盖
* 读到没有意义的数据，因为只写入了一半
* 竞态条件导致bug

事务简化上面的问题，整个事务要么成功，要么失败。

不是所有应用都需要事务， 有时削弱事务，或者去掉事务，得到更高的性能，更高的可用性

一些安全的特性，没有事务也可以达到，


## The Slippery Concept of a Transaction

​现今，几乎所有的关系型数据库和一些非关系数据库都支持事务

事务有其优势和局限性



#### The Meaning of ACID
* atomicity 原子性

ACID原子性的定义特征是：能够在错误时中止事务，丢弃该事务进行的所有写入变更的能力。 或许 可中止性（abortability） 是更好的术语，



* consistency 一致性

一致性这词有很多相似的词：
* 最终一致性：异步复制系统中的最终一致性问题
* 一致性哈希：是某些系统用于重新分区的一种分区方法。
* 在CAP中 一致性是  linearizability ， 可线性化。
* 在ACID中 一致性是 数据库在良好状态 ， 事务开始结束，数据库的完整性不受破坏

比如数据库有约束，钱总数是零，且应用操作的时候，保证这个规则， 那么任何时刻，钱总数都是零，这就是 一致性.

ACID一致性的概念是，对数据的一组特定陈述必须始终成立。即不变量（invariants）


原子性 隔离性 持久性是 数据库的特性，一致性是应用的特性。应用依赖 数据库的特性， 实现 一致性，


* isolation：隔离性

每个事务可以假装它是唯一在整个数据库上运行的事务。数据库确保当事务已经提交时，结果与它们按顺序运行（一个接一个）是一样的，尽管实际上它们可能是并发运行的



* durability  持久性

一旦事务成功完成，即使发生硬件故障或数据库崩溃，写入的任何数据也不会丢失。



#### Single-Object and Multi-Object Operations

一个事务读取另一个事务的未被执行的写入（“脏读”）。脏读违反隔离性

Single-object writes

当单个对象发生改变时，原子性和隔离也是适用的。

The need for multi-object transactions

许多分布式数据存储已经放弃了多对象事务，因为多对象事务很难跨分区实现，而且在需要高可用性或高性能的情况下，它们可能会碍事。但说到底，在分布式数据库中实现事务，并没有什么根本性的障碍


为什么需要多对象事务？
* 在关系数据模型中，一个表中的行通常具有对另一个表中的行的外键引用。
* 在具有二级索引的数据库中，每次更改值时都需要更新索引。从事务角度来看，这些索引是不同的数据库对象



Handling errors and aborts

事务的一个关键特性是，如果发生错误，它可以中止并安全地重试。 

ACID数据库基于这样的哲学：如果数据库有违反其原子性，隔离性或持久性的危险，则宁愿完全放弃事务，而不是留下半成品。


尽管重试一个中止的事务是一个简单而有效的错误处理机制，但它并不完美：


* 如果事务实际上成功了，但是在服务器试图向客户端确认提交成功时网络发生故障（所以客户端认为提交失败了），那么重试事务会导致事务被执行两次——除非你有一个额外的应用级除重机制。
* 如果错误是由于负载过大造成的，则重试事务将使问题变得更糟，而不是更好。为了避免这种正反馈循环，可以限制重试次数，使用指数退避算法，并单独处理与过载相关的错误（如果允许）。
* 仅在临时性错误（例如，由于死锁，异常情况，临时性网络中断和故障切换）后才值得重试。在发生永久性错误（例如，违反约束）之后重试是毫无意义的。
* 如果事务在数据库之外也有副作用，即使事务被中止，也可能发生这些副作用。例如，如果你正在发送电子邮件，那你肯定不希望每次重试事务时都重新发送电子邮件。如果你想确保几个不同的系统一起提交或放弃，二阶段提交（2PC, two-phase commit） 可以提供帮助（“原子提交和两阶段提交（2PC）”中将讨论这个问题）。
* 如果客户端进程在重试中失效，任何试图写入数据库的数据都将丢失。



## Weak Isolation Levels 


数据库一直试图通过提供事务隔离（transaction isolation） 来隐藏应用程序开发者的并发问题。

从理论上讲，隔离可以通过假装没有并发发生，让你的生活更加轻松：

可序列化（serializable） 的隔离等级意味着数据库保证事务的效果与连续运行（即一次一个，没有任何并发）是一样的。

可序列化 会有性能损失，许多数据库不愿意支付这个代价【8】。

因此，系统通常使用较弱的隔离级别来防止一部分，而不是全部的并发问题。

这些隔离级别难以理解，并且会导致微妙的错误，但是它们仍然在实践中被使用【23】。



#### Read Committed

读已提交：
* 读数据库的时候，只能读到已经提交的数据，  没有脏读  dirty read  
* 写数据库时，只会覆盖已经提交的数据，没有脏写 dirty write 

No dirty reads

脏读：
一个事务更新了数据，但是还没有提交或中止，另一个事务能看到未提交的数据

![](/img/ddia/fig7-2.png)



No dirty writes
脏写：

后面的写入会覆盖一个尚未提交的值。这被称作脏写（dirty write）

![](/img/ddia/fig7-5.png)

读已提交，必须防止脏写，通常延迟第二个写，等第一个写事务提交或放弃


Implementing read committed

防止脏写：
* 当在事务里面，需要修改一个数据的时候，先加一个锁，直到事务结束，才释放。如果第一个事务对一个数据加了锁，第二个事务想写这个数据，就拿不到锁，这样就防止了脏写。

防止脏读：
* 选项一：还是用之前的锁，写之前加锁，读的时候等锁释放 再读，但是性能不好
* 选项二：事务写值，保留旧值，读的时候读旧值


#### Snapshot Isolation and Repeatable Read

读已提交会出现读偏斜的问题

![](/img/ddia/fig7-6.png)

这种情况称为 读偏斜  Read skew  或者  不可重复读， nonrepearable read 

读偏斜某些情况下不能容忍：
* 备份的时候，各种事务正常进行，备份读到的数据，有的是新的，有的是旧的，不一致的问题就出现了
* 分析数据时，如果有读偏斜，分析出来的数据没有意义


快照隔离可以解决这个问题，每个事务都读数据库的一个一致的快照

在事务内看到的数据，都是事务开始时 已经提交的事务的 一致的快照

对于备份和分析，很适合 快照隔离级别， 


Implementing snapshot isolation


为了实现快照隔离，数据库使用了我们看到的用于防止图7-4中的脏读的机制的一般化。

数据库必须可能保留一个对象的几个不同的提交版本，因为各种正在进行的事务可能需要看到数据库在不同的时间点的状态。

因为它并排维护着多个版本的对象，所以这种技术被称为多版本并发控制（MVCC, multi-version concurrentcy control）。


Visibility rules for observing a consistent snapshot 观察一致性快照的可见性规则

当一个事务从数据库中读取时，事务ID用于决定它可以看见哪些对象，看不见哪些对象。通过仔细定义可见性规则，数据库可以向应用程序呈现一致的数据库快照。工作如下：


* 在每次事务开始时，数据库列出当时所有其他（尚未提交或中止）的事务清单，即使之后提交了，这些事务的写入也都会被忽略。
* 被中止事务所执行的任何写入都将被忽略。
* 由具有较晚事务ID（即，在当前事务开始之后开始的）的事务所做的任何写入都被忽略，而不管这些事务是否已经提交。
* 所有其他写入，对应用都是可见的。


Indexes and snapshot isolation 索引和 快照隔离


Repeatable read and naming confusion

快照隔离是一个有用的隔离级别，特别对于只读事务而言。

但是，许多数据库实现了它，却用不同的名字来称呼。

在Oracle中称为可序列化（Serializable）的

在PostgreSQL和MySQL中称为可重复读（repeatable read）【23】。




#### Preventing Lost Updates


并发的写入事务之间还有其他几种有趣的冲突。其中最着名的是丢失更新（lost update） 问题

更新丢失会出现在这种场景：计数器。先读，然后根据读到的值，加一，写入

这是常见的问题，很多解决办法发展出来：

* Atomic write operations 原子写
UPDATE counters SET value = value + 1 WHERE key = 'foo';

原子操作实现：读的时候加锁，不然其他人读


* Explicit locking 显式锁定
读的时候加锁

select for update  数据库锁定所有返回的行 

* Automatically detecting lost updates
自动检测丢失的更新

MySQL InnoDB’s repeatable read does not detect lost updates [23]


* Compare-and-set

* Conflict resolution and replication
在复制的数据库中， 数据有副本，并发更新时，可能是不同节点上， 要想防止更新丢失，需要额外的机制




#### Write Skew and Phantoms

Characterizing write skew

写偏斜：

两个事务同时读多个对象，然后各更新其中不同的几个，可能发生写偏斜




防止写偏斜：
* 防止写偏斜需要多个对象的原子操作

* 显式锁定所有依赖的数据



More examples of write skew


Phantoms causing write skew

phantom 幻读：一个事务中的写入改变另一个事务的搜索查询的结果，被称为幻读【3】




Materializing conflicts

如果幻读的问题是没有对象可以加锁，也许可以人为地在数据库中引入一个锁对象？

使用 SELECT FOR UPDATE 对选择的所有数据加锁，这种方法被称为物化冲突（materializing conflicts）

除了这种方式，还可以用可序列化解决这个问题。



## Serializability

可序列化（Serializability）隔离通常被认为是最强的隔离级别。

它保证即使事务可以并行执行，最终的结果也是一样的，就好像它们没有任何并发性，连续挨个执行一样。

因此数据库保证，如果事务在单独运行时正常运行，则它们在并发运行时继续保持正确 —— 

换句话说，数据库可以防止所有可能的竞争条件。


#### Actual Serial Execution

避免并发问题的最简单方法就是完全不要并发

可序列化：在单个线程上按顺序一次只执行一个事务。


为什么07年才提出这种隔离级别？
* ram足够便宜，所有进程都放在内存上
* tp事务较短，ap事务不用tp事务的线程

为了充分利用单一线程，需要与传统形式不同的结构的事务。


Encapsulating transactions in stored procedures

如果使用可序列化隔离级别，如果一个事务需要用户多次输入，那么系统的吞吐会非常低

​出于这个原因，具有单线程串行事务处理的系统不允许交互式的多语句事务。

取而代之，应用程序必须提前将整个事务代码作为存储过程提交给数据库。



Pros and cons of stored procedures

缺点：
* 每种数据库 存储过程的语法都不相同。没有跟上编程语言的发展，丑陋陈旧，没有相关库
* 管理，调试，版本控制，部署，测试，监控 很尴尬


上面的缺点可以克服，现代的存储过程使用现有的编程语言， 例如 Redis使用Lua。

存储过程与内存存储，使得在单个线程上执行所有事务变得可行。

由于不需要等待I/O，且避免了并发控制机制的开销，它们可以在单个线程上实现相当好的吞吐量。



Partitioning

顺序执行所有事务使并发控制简单多了，但数据库的事务吞吐量被限制为单机单核的速度。

对于写入吞吐量较高的应用，单线程事务处理器可能成为一个严重的瓶颈。

为了扩展到多个CPU核心和多个节点，可以对数据进行分区

如果每个事务只需要在单个分区中读写数据

在这种情况下可以为每个分区指派一个独立的CPU核，事务吞吐量就可以与CPU核数保持线性扩展【47】。

但是，对于需要访问多个分区的任何事务，数据库必须在触及的所有分区之间协调事务。存储过程需要跨越所有分区锁定执行，以确保整个系统的可串行性。

由于跨分区事务具有额外的协调开销，所以它们比单分区事务慢得多。



Summary of serial execution

可序列化特点：

* 每个事务都必须小而快，只要有一个缓慢的事务，就会拖慢所有事务处理。
* 仅限于活跃数据集可以放入内存的情况。很少访问的数据可能会被移动到磁盘，但如果需要在单线程执行的事务中访问，系统就会变得非常慢x。
* 写入吞吐量必须低到能在单个CPU核上处理，如若不然，事务需要能划分至单个分区，且不需要跨分区协调。
* 跨分区事务是可能的，但是它们的使用程度有很大的限制。



#### Two-Phase Locking (2PL)

大约30年来，在数据库中只有一种广泛使用的序列化算法：两阶段锁定（2PL，two-phase locking）


#### Serializable Snapshot Isolation (SSI)

一方面，我们实现了性能不好（2PL）或者扩展性不好（串行执行）的可序列化隔离级别。

另一方面，我们有性能良好的弱隔离级别，但容易出现各种竞争条件（丢失更新，写入偏差，幻读等）。

序列化的隔离级别和高性能是从根本上相互矛盾的吗？

一个称为可序列化快照隔离（SSI, serializable snapshot isolation） 的算法是非常有前途的。

它提供了完整的可序列化隔离级别，但与快照隔离相比只有只有很小的性能损失。 

SSI是相当新的：它在2008年首次被描述【40】，并且是Michael Cahill的博士论文【51】的主题。
